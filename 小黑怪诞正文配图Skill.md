---
技能名称: 小黑怪诞正文配图Skill
技能描述: 中文文章正文配图与小黑怪诞手绘插图生成技能。用于把中文文章、帖子、博客、方法论、AI 工作流文档、单个观点、流程结构、状态变化或概念隐喻，转成 Ian Xiaohei 风格的 16:9 白底手绘正文配图。默认使用小黑 IP、纯白背景、黑色手绘线稿、大量留白、少量红橙蓝中文手写批注。用户说“小黑配图”“怪诞正文配图”“给文章做配图”“生成文章插图”“shot list”“去掉图中文字标题”“增强小黑参与感”时使用。图片默认使用 4.7 模型的 2k 版；用户明确指定模型版本时按用户指定。
---

技能内容：

分为**交互阶段**和**执行阶段**。交互阶段把用户正文、观点或参考图规范化为配图 brief；执行阶段按“正文消化 → 认知锚点 → shot list → 单张 prompt → 图片生成 / 改图 → QA → 必要修复 → 交付”的链路执行。

本技能改编自 Ian Xiaohei Illustrations 的视觉方法。生成时保留 Ian / 小黑配图的风格归属与命名，不把小黑改造成通用吉祥物、普通卡通角色或装饰符号。

默认模型要求：

- 图片生成默认使用 4.7 模型的 2k 版。
- 用户明确指定图片模型版本时，按用户指定执行。
- 本技能不包含视频生成主流程；用户要求把配图继续做成视频时，先完成本文配图，再交给视频类技能处理。
- 不输出本地命令、任务查询命令、下载命令或工程路径。

## 一、触发范围

### 1.1 应使用本技能的情况

- 用户给出中文文章、段落、帖子、博客、观点、方法论、AI 工作流、产品思考、组织管理内容、抽象判断或课程内容，希望生成正文配图。
- 用户要求“小黑配图”“怪诞手绘”“白底黑线”“正文插图”“文章插图”“配几张图”“给每个段落配图”“生成 shot list”。
- 用户上传旧图、草图、截图或参考图，希望改成小黑怪诞正文配图。
- 用户要求去掉图中大标题、减少文字、增强小黑参与感、把正式图改得更怪诞、更松弛、更像正文配图。
- 用户只要配图策划，不要求生成图片；此时输出 shot list 和每张图的 prompt。

### 1.2 不应使用本技能的情况

- 用户要写实摄影、电商主图、品牌 KV、PPT 信息图、正式流程图、App 截图或复杂数据可视化。
- 用户要复刻某张历史案例的构图、角色动作或具体创意。只能继承视觉 DNA，不复制旧图。
- 用户要求生成误导性事实、侵权商标、现实人物冒充、违法危险内容或其他不适合生成的内容。
- 用户主要目标是视频成片、连续分镜视频或长片剪辑；此时只可先生成静态配图，再交给视频技能。

## 二、工具映射

| 工具名 | 本技能用途 | 下一步 |
|---|---|---|
| `generate_form_for_info_collection` | 当正文、主题、目标张数或改图目标缺失且无法推断时收集信息 | 表单完成后进入正文消化 |
| `creation_agent_search` | 可选：用户只给公开主题、热点或行业背景时搜索辅助语境 | 搜索结果只作理解背景，必须重新发明原创隐喻 |
| `text2image` | 无参考图时逐张生成小黑正文配图 | 每张图进入 QA |
| `image2image` | 基于用户上传旧图、草图、参考图进行改图、去标题、重构或增强小黑 | 编辑后进入 QA |
| `image_super_resolution` | 结构、风格和文字密度均合格后高清化 | 高清图进入最终交付 |

在线版适配：

- 生成图片时显式选择 4.7 模型的 2k 版；不要使用自动升级模型，除非用户明确指定。
- 每张配图独立生成，不把多张文章配图拼进同一张大图。
- 中文手写标注必须短、少、可读；若文字容易乱码，优先减少标注数量或用简单箭头、圈注、短词替代。

## 三、交互阶段

### 3.1 信息提取

先从用户输入、上传图片和上下文中提取字段。

| 字段 | 必填 | 默认值 | 缺失时处理 |
|---|---|---|---|
| 正文 / 观点 / 主题 | 是 | 无 | 缺失且无法从上下文推断时，调用 `generate_form_for_info_collection` |
| 任务类型 | 是 | 自动推断 | `shot_list_only` / `article_illustration` / `single_concept` / `image_edit` |
| 目标张数 | 否 | 单观点 1 张；短文 1-3 张；普通文章 4-8 张；长文 6-9 张 | 用户指定 N 张则按 N；不得平均铺满无意义段落 |
| 是否生成图片 | 否 | 用户说“生成、做图、输出图片”则生成 | 用户只要策略、shot list 或 prompt 时不调用图片工具 |
| 参考图 / 旧图 | 否 | 无 | 有图走 `image2image`，并识别需要保留和需要改掉的部分 |
| 图片比例 | 否 | 16:9 横版 | 用户明确指定时覆盖 |
| 图片模型 | 否 | 4.7 模型的 2k 版 | 用户指定时覆盖 |
| 中文标注密度 | 否 | 最多 5-8 处，每处 2-8 字 | 用户要少字时进一步压缩到 0-3 处 |
| 输出用途 | 否 | 正文插图 | 可推断为公众号、博客、课程页、长文、社媒帖、产品思考等 |
| 文字标题策略 | 否 | 不写大标题 | 只允许少量贴近画面的小标签和箭头批注 |

### 3.2 必要追问规则

只在以下情况追问：

- 没有正文、观点、主题，也无法从上传图推断配图对象。
- 用户要求为特定品牌、课程或产品配图，但没有给出必须保留的准确名称或事实。
- 用户上传旧图要求改图，但没有说明要保留什么、改掉什么，且图片意图不清。
- 用户要求严格数量、平台尺寸或特殊文字内容，但关键信息缺失。

其他字段使用默认值继续执行。

## 四、正文消化与认知锚点

### 4.1 消化原则

正文配图不是逐段翻译，也不是把文章观点写进图里。先抽取值得被视觉化的“认知锚点”：

- 反直觉判断：读者原本以为 A，文章指出 B。
- 状态变化：从混乱到清晰，从堵住到流动，从脆弱到稳定。
- 结构关系：输入、处理、输出；人、系统、工具；表层现象、底层机制。
- 冲突张力：旧习惯和新方法互相拉扯。
- 使用场景：人在系统中如何移动、选择、等待、误解、修复。
- 方法动作：拆解、绕开、连接、压缩、筛选、校准、复盘。
- 关键比喻：把抽象概念变成一个低技术、可手画、略怪诞的物件或动作。

### 4.2 锚点选择规则

- 每张图只表达一个核心结构、状态或隐喻。
- 优先选择“能让读者在正文中停一下”的位置，不平均覆盖所有段落。
- 同一篇文章中，不重复使用同一种构图结构。
- 不把抽象词直接画成大字牌；要把概念转化为可见动作。
- 如果某个锚点只能靠长文字解释，说明它不适合做图，需要重新选锚点。

### 4.3 锚点输出格式

```Markdown
## 认知锚点

| anchor_id | 正文位置 | 核心意思 | 视觉化机会 | 风险 |
|---|---|---|---|---|
| A01 | 开头 / 中段 / 结尾 |  |  |  |
```

## 五、Shot List 规划

### 5.1 数量规则

| 内容类型 | 默认张数 | 说明 |
|---|---:|---|
| 单个观点 / 金句 / 一句话需求 | 1 | 做一张强概念图 |
| 短文 / 帖子 / 轻量方法论 | 1-3 | 覆盖开头钩子、核心方法、结尾转折 |
| 普通长文 / 产品思考 / 工作流文档 | 4-8 | 每张图服务一个认知锚点 |
| 很长的系列文章 / 课程稿 | 6-9 | 只挑最高价值节点，不逐段铺图 |
| 用户指定 N 张 | N | 按 N 个独立图号执行，数量必须对齐 |

### 5.2 Shot List 输出格式

```Markdown
## 小黑配图 Shot List

| image_id | 放置位置 | 图名 | 核心意思 | 结构类型 | 小黑动作 | 画面元素 | 中文标注 | 生成方式 |
|---|---|---|---|---|---|---|---|---|
| XH-IMG01 | 开头后 |  |  | concept_metaphor |  |  |  | text2image |
```

### 5.3 执行规则

- 用户只要策略、shot list 或 prompt：输出 shot list 后停止。
- 用户要求生成图片：shot list 只是中间产物，必须继续生成图片。
- 每个 `image_id` 独立构建 prompt、独立调用工具、独立 QA。
- 如果某张图不合格，只重做该 `image_id`，不打乱其他已合格图片。

## 六、构图类型库

| structure_type | 适用内容 | 画面结构 | 小黑角色 |
|---|---|---|---|
| `workflow` | 步骤、链路、AI 流程、协作流 | 左到右或环形路径，少量节点 | 小黑搬运、连接、踩开关、排队、修管道 |
| `system_parts` | 系统构成、角色关系、模块解释 | 一个怪异机器或剖面结构 | 小黑钻进系统内部检修 |
| `before_after` | 变化、升级、对比、误区纠正 | 左右对比或上下分层 | 小黑从旧状态逃到新状态 |
| `character_state` | 心智负担、焦虑、卡住、顿悟 | 小黑处在夸张但克制的状态中 | 小黑承受、观察、躲避、推开障碍 |
| `concept_metaphor` | 抽象观点、方法论、反直觉判断 | 一个低技术怪物件或荒诞道具 | 小黑操作该道具，使概念成立 |
| `method_layers` | 框架、原则、层级、策略 | 叠层、剖面、阶梯、篮筐、过滤器 | 小黑在层与层之间穿行 |
| `map_route` | 选择路径、路线、决策、迁移 | 地图、岔路、管线、岛屿 | 小黑沿路径移动，留下痕迹 |
| `mini_comic` | 小故事、连续动作、轻剧情 | 2-4 个小分区，但仍是一张 16:9 图 | 小黑完成一个短动作链 |

构图选择规则：

- 正文讲“怎么做” → 优先 `workflow` 或 `method_layers`。
- 正文讲“系统为什么这样” → 优先 `system_parts`。
- 正文讲“以前和现在不同” → 优先 `before_after`。
- 正文讲“人的状态” → 优先 `character_state`。
- 正文讲“抽象洞察” → 优先 `concept_metaphor`。
- 正文讲“选择路线” → 优先 `map_route`。
- 同一组配图中尽量避免连续两张使用相同 structure_type。

## 七、视觉 DNA

### 7.1 必须保留

- 16:9 横版，纯白背景。
- 黑色手绘线稿，像随手画在白纸上的产品思考草图。
- 大量留白，主体通常占画面 40%-60%，不要铺满。
- 线条有轻微手绘抖动，但整体清楚、克制、干净。
- 画面可以荒诞，但信息结构必须清楚。
- 少量红、橙、蓝手写批注：
  - 红色用于问题、危险、重点、断点；
  - 橙色用于主路径、行动方向、当前焦点；
  - 蓝色用于辅助说明、背景状态、次要节点。
- 标注短而少，每处 2-8 字，最多 5-8 处。
- 画面中的箭头、圈注、虚线、钉子、管道、开关、标签都应像手绘标记。

### 7.2 必须避免

- 不要做成 PPT、正式流程图、商业信息图、咨询公司图表。
- 不要做成儿童绘本、圆润卡通、吉祥物海报、品牌 IP 海报。
- 不要做成复杂漫画页、厚涂插画、3D 渲染、真实摄影、赛博 UI。
- 不要大面积填色，不要渐变背景，不要装饰性纹理。
- 不要在左上角写大标题，例如“Workflow”“系统架构图”“常见问题”“路线图”。
- 不要把正文原句大段搬进画面。
- 不要生成水印、二维码、真实界面截图、复杂表格、长段文字。

## 八、小黑 IP 规则

### 8.1 外观

- 小黑是一个小型黑色实心角色，身体像一团简化的黑色块状生物。
- 眼睛是白色小圆点或小椭圆点。
- 四肢极简，细线脚或短线手即可。
- 表情冷静、困惑、笨拙、认真或轻微无语，不做夸张卖萌表情。
- 不加衣服、头发、复杂五官、品牌装饰或拟人化职业制服，除非用户明确要求且不破坏识别。

### 8.2 角色功能

- 小黑必须承担核心动作，而不是站在旁边装饰。
- 如果去掉小黑后画面仍然完全成立，则该图失败。
- 小黑可以推、拉、搬、举、钻、测量、连接、剪断、等待、躲避、修补、观察、标记。
- 小黑的动作要让抽象概念被看见，例如“把混乱线团接到一个小机器里”“在巨大漏斗下接住少量有效信号”。
- 小黑不解释观点，不举长牌子，不当主持人，不在图中说大段话。

## 九、原创隐喻生成

### 9.1 隐喻原则

- 把抽象概念变成低技术、可手绘、略怪诞但清楚的物件。
- 优先使用日常物件、简陋机器、管道、漏斗、梯子、岛屿、洞口、抽屉、橡皮筋、磁铁、开关、传送带、拼图、鱼缸、篮筐、便签、桥、迷宫。
- 物件不需要真实可用，但逻辑要能被一眼理解。
- 怪诞来自“物件关系不寻常”，不是来自复杂细节。
- 每张图只允许一个主隐喻，辅助元素最多 3-5 个。

### 9.2 反复刻规则

- 不复刻已有案例中的具体构图、角色站位、道具组合或文字标签。
- 不把用户给的例图当作可复制模板；只提取风格、留白、线条、标注密度和角色功能。
- 同一篇文章中，如果两个锚点都想用桥、漏斗或管道，必须替换其中一个隐喻。
- 生成前检查是否出现陈词滥调：灯泡代表灵感、拼图代表协作、齿轮代表系统、火箭代表增长。能替换则替换。

## 十、Prompt 构建

### 10.1 单张生成 Prompt 模板

为每个 `image_id` 单独构建 prompt，不要把整组配图塞进一条 prompt。

```Markdown
生成一张独立的 16:9 横版中文文章正文配图。

模型：默认使用 4.7 模型的 2k 版；如果用户明确指定其他模型，按用户指定。

用途：这张图用于正文中段配图，不是封面，不是海报，不是 PPT。

视觉 DNA：
- 纯白背景；
- 黑色手绘线稿；
- 大量留白；
- 轻微手绘抖动，但结构清楚；
- 稀疏红 / 橙 / 蓝手写批注；
- 整体像一个聪明但怪诞的手绘产品思考草图。

固定角色：小黑。
- 小黑是一个小型黑色实心角色，白色小圆点眼睛，极简细线四肢；
- 小黑表情冷静、笨拙、认真；
- 小黑必须执行核心动作，不要只站在旁边。

正文锚点：
- image_id: {image_id}
- 核心意思: {core_idea}
- 结构类型: {structure_type}
- 读者应看懂的关系: {reader_takeaway}

画面设计：
- 画面中心是 {main_metaphor}；
- 小黑正在 {xiaohei_action}；
- 主要元素包括 {visual_elements}；
- 元素之间的关系是 {relationship_logic}；
- 构图保持简洁，主体约占画面 40%-60%；
- 留出白色边界，不铺满画面。

中文手写标注：
- 只放 {label_count} 个短标签；
- 每个标签 2-8 个字；
- 可用标签：{labels}
- 标签靠近相关元素，不要写大标题，不要写长句。

颜色规则：
- 黑色线稿为主；
- 红色只标记问题或断点；
- 橙色只标记主路径或当前行动；
- 蓝色只标记辅助状态；
- 不使用其他大面积颜色。

负向约束：
- 不要 PPT 信息图；
- 不要正式流程图；
- 不要商业插画；
- 不要儿童卡通；
- 不要 3D 渲染；
- 不要真实摄影；
- 不要科技 UI；
- 不要大标题；
- 不要长段文字；
- 不要水印、二维码、logo；
- 不要复刻旧案例构图。
```

### 10.2 改图 Prompt：去掉大标题

用户上传一张基本合格但带有左上角标题或多余文字的图时，使用 `image2image`：

```Markdown
基于用户上传图片进行小幅编辑，保持原有 16:9 白底黑色手绘线稿风格和小黑角色。

编辑目标：
- 移除左上角或画面顶部的大标题；
- 移除过长、过密或像 PPT 标题的文字；
- 保留少量贴近元素的短手写标注；
- 保留小黑、主隐喻、画面结构和留白；
- 不新增复杂背景，不改变整体构图。

输出仍然是小黑怪诞正文配图：白底、黑线、少量红橙蓝批注、没有封面标题。
```

### 10.3 改图 Prompt：增强小黑参与感

用户上传图中小黑只是装饰或不存在时，使用 `image2image`：

```Markdown
基于用户上传图片重构为小黑怪诞正文配图。

保留：
- 原图的核心概念；
- 主要隐喻或结构；
- 16:9 白底黑色手绘线稿；
- 大量留白和少量红橙蓝手写批注。

增强：
- 加入或强化小黑角色；
- 小黑必须执行核心动作：{xiaohei_action}；
- 让小黑与主隐喻发生物理关系，例如推、拉、钻入、连接、测量、剪断、修补或躲避；
- 如果没有小黑，画面就不应成立。

避免：
- 不要把小黑做成吉祥物海报；
- 不要新增大标题；
- 不要做成儿童卡通；
- 不要让画面变复杂。
```

### 10.4 改图 Prompt：正式图转怪诞正文配图

用户上传 PPT、流程图、正式信息图或截图，希望转成小黑风格时，使用 `image2image` 或重新 `text2image`：

```Markdown
把用户上传的正式图转换为一张小黑怪诞正文配图。

只保留原图最核心的结构关系：{core_structure}。

重新绘制为：
- 16:9 纯白背景；
- 黑色手绘线稿；
- 大量留白；
- 一个低技术、略怪诞但清楚的主隐喻：{main_metaphor}；
- 小黑正在执行核心动作：{xiaohei_action}；
- 最多 5 个短中文手写标注；
- 少量红橙蓝圈注和箭头。

不要保留原图的 PPT 样式、框线表格、正式图标、UI 截图、密集文字和商业配色。
```

## 十一、执行阶段

### 11.1 任务路由

| task_type | 条件 | 执行 |
|---|---|---|
| `shot_list_only` | 用户只要规划、选题、shot list 或 prompt | 输出 brief、认知锚点、shot list、每张 prompt |
| `single_concept` | 用户只给一个观点或一句话，且要求生成图 | 规划 1 张图，调用 `text2image` 或 `image2image` |
| `article_illustration` | 用户给文章并要求配多张图 | 先输出 shot list，再逐张生成 |
| `image_edit` | 用户上传旧图并要求去标题、改风格、增强小黑 | 调用 `image2image`，必要时重新生成 |

### 11.2 无参考图生成

当没有参考图或旧图时：

1. 生成 brief。
2. 提取认知锚点。
3. 输出 shot list。
4. 用户要求生成图片时，为每个 `image_id` 构建 prompt。
5. 调用 `text2image`：
   - `prompt`: 当前 `image_id` 的完整 prompt；
   - `aspect_ratio`: `16:9`；
   - `model`: `4.7 模型的 2k 版`，除非用户明确指定；
   - `n`: 1。
6. 输出进入 QA。
7. QA 合格后交付；清晰度不足但结构合格时，调用 `image_super_resolution`。

### 11.3 有参考图或旧图改图

当用户上传图片时：

1. 分析图片角色：
   - 旧小黑图；
   - 草图；
   - 正式图 / PPT / 流程图；
   - 文章截图；
   - 风格参考；
   - 必须保留的主体。
2. 提取 `required_preserve` 和 `required_change`。
3. 构建改图 prompt。
4. 调用 `image2image`：
   - `images`: 用户上传图；
   - `prompt`: 改图 prompt；
   - `aspect_ratio`: `16:9`；
   - `model`: `4.7 模型的 2k 版`，除非用户明确指定；
   - `n`: 1。
5. 输出进入 QA。
6. 如果编辑损坏了核心概念，回到 prompt 修复；如果只是清晰度不足，调用 `image_super_resolution`。

### 11.4 多张文章配图

多张图执行时建立 manifest：

```Markdown
## 小黑配图 Manifest

| image_id | status | tool | model | aspect_ratio | qa_score | next_action |
|---|---|---|---|---|---|---|
| XH-IMG01 | planned | text2image | 4.7 模型的 2k 版 | 16:9 |  |  |
```

状态规则：

- `planned`：已规划，未调用工具。
- `generated`：已生成，未 QA。
- `needs_fix`：QA 未通过，需要修复。
- `passed`：QA 通过，可交付。
- `failed`：多次修复仍失败，保留失败原因。

数量规则：

- 用户要 N 张，manifest 必须有 N 个 `image_id`。
- 最终交付必须说明 `requested_count`、`passed_count`、`failed_count`。
- 不得用一张拼图代替 N 张独立配图。

## 十二、质量门

### 12.1 评分标准

| 分数 | 判断 | 行动 |
|---:|---|---|
| 0-59 | 不可用 | 重写 prompt 或换隐喻 |
| 60-74 | 大修 | 保留少量有效部分，重生成 |
| 75-84 | 小修可测 | 针对失败点改图 |
| 85-94 | 可生产 | 可交付，必要时高清化 |
| 95-100 | 样板级 | 可作为同组风格参考 |

### 12.2 Prompt QA

生成或改图前检查 prompt：

- 是否明确 16:9 横版、纯白背景、黑色手绘线稿。
- 是否明确使用小黑，并且小黑承担核心动作。
- 是否只有一个核心隐喻或结构。
- 是否有具体画面元素，而不是抽象形容词堆叠。
- 是否限制短中文标注数量。
- 是否写明不要大标题、PPT、正式流程图、商业插画、儿童卡通、3D、真实摄影。
- 是否避免复刻旧案例构图。

Prompt QA 低于 85 分时，不得调用图片工具。

### 12.3 图片 QA

生成后检查：

- 画幅是否为 16:9 横版。
- 背景是否纯白或接近纯白。
- 线条是否为黑色手绘线稿。
- 小黑是否存在，且承担核心动作。
- 去掉小黑后画面是否仍成立；如果成立，则失败。
- 画面是否只有一个主要结构或隐喻。
- 留白是否充足。
- 中文标注是否短、少、可读。
- 是否出现大标题、长段文字、水印、二维码、logo、真实 UI、PPT 样式。
- 是否有过度卡通化、商业插画化、厚涂化、3D 化。
- 是否复刻旧案例构图。

图片 QA 低于 80 分时，不得交付为最终图。

## 十三、失败类型与修复

| 失败类型 | 表现 | 修复动作 |
|---|---|---|
| `too_ppt` | 像正式流程图、咨询图、PPT | 改用低技术隐喻，减少框线和节点 |
| `xiaohei_decorative` | 小黑只是站着或不存在 | 强化小黑核心动作，要求去掉小黑则图不成立 |
| `too_much_text` | 文字太多、像标题页 | 去掉大标题，保留 0-5 个短标签 |
| `not_white_background` | 背景复杂或有渐变 | 强制纯白背景、大量留白 |
| `too_cute` | 儿童卡通、萌系吉祥物 | 改为克制、笨拙、冷静的小黑 |
| `too_polished` | 商业插画、厚涂、3D | 强化黑线手绘、草图感、低技术道具 |
| `metaphor_unclear` | 看不出核心关系 | 减少元素，改成更直接的动作关系 |
| `copied_example` | 明显复刻旧案例构图 | 重新发明隐喻，只保留风格 DNA |
| `bad_chinese_text` | 中文乱码或不可读 | 减少文字，改为短词、箭头、圈注 |

修复规则：

- 失败来自构图或隐喻：重写 prompt 后重新 `text2image`。
- 失败来自标题、文字、局部风格：优先 `image2image` 局部改图。
- 失败来自清晰度：只有结构和风格合格后才 `image_super_resolution`。
- 同一张图连续两次因同一问题失败时，必须换隐喻或换构图类型。
