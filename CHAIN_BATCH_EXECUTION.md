# Chain, Batch, and Concurrency Contract

本文件是 `dreamina-skills-online` 所有复合型即梦在线 skill 的共享执行规范。目标是让链式调用更稳定、批量任务数量正确、并发生成更快，同时避免依赖关系断裂。

## 1. 先建任务清单

任何多步骤或批量任务，都必须先生成一个任务清单，再调用工具。

任务清单至少包含：

| 字段 | 说明 |
|---|---|
| `job_id` | 稳定编号，例如 `SKU01-IMG03`、`R08`、`TS-S04` |
| `target` | 当前任务要产出的内容 |
| `depends_on` | 依赖的上游任务；无依赖则为空 |
| `tool` | 即梦在线工具名 |
| `model` | 模型版本；图片默认 4.7 模型 2k 版，视频默认 `seedance2.0fast_vip`，用户指定版本时按用户指定 |
| `input_refs` | 输入参考图、脚本、关键帧或上游结果 |
| `status` | `planned`、`running`、`done`、`failed`、`retrying` |
| `retry_count` | 重试次数 |
| `quality_gate` | 当前任务必须通过的检查 |

不要只写自然语言计划。必须有可点数、可追踪的 manifest。

## 2. 数量守恒

用户给出数量时，数量是承诺，不自动缩减。

| 场景 | 数量规则 |
|---|---|
| 设计 N 款 | 必须有 N 个 `design_id`，每个设计至少 1 张图或 1 个失败记录 |
| N 个 SKU 电商素材 | 每个 SKU 都要有完整预设槽位，例如 Amazon 每 SKU 9 图 + 1 视频 + 文案 |
| 脱口秀分段 | 脚本确认后分段数固定，必须生成同样数量的视频 segment |
| 多平台交付 | 平台数 × 每平台槽位数都要进入任务清单 |

最终输出必须报告：

- `requested_count`
- `planned_count`
- `done_count`
- `failed_count`
- `missing_count`

如果 `missing_count > 0`，不得声称完整完成。

## 3. 依赖优先

只有依赖完成后才能进入下游任务。

常见依赖：

- 角色/产品/设计身份锁 → 图片生成。
- 产品母图或用户上传图 → 电商槽位图。
- 脚本确认 → 脱口秀分段视频。
- 分段视频全部完成 → `video_editor` 拼接。
- 主图/微距图/佩戴图完成 → 多模态视频生成。

不要为了并发跳过身份锁、脚本确认、母图确认或必要参考图。

## 4. 并发 fanout

可以并发的是同一层级、互不依赖的任务。

适合并发：

- 同一批设计图生成。
- 同一 SKU 的多个独立电商槽位图。
- 多个 SKU 的同类槽位图。
- 脚本确认后的多个脱口秀视频分段。
- 已生成素材的高清放大。

不适合并发：

- 脚本未确认前生成视频。
- 产品母图未通过前生成全部槽位图。
- 分段视频未完成前做最终拼接。
- 质量检查未完成前将失败图当作下游参考。

默认并发建议：

| 任务类型 | 默认并发 |
|---|---|
| 图片生成 | 4-6 |
| 图片高清/抠图 | 6-8 |
| 视频分段生成 | 2-4 |
| 最终 `video_editor` | 1 |

如果在线环境限制并发，则按平台上限执行，但仍保留完整任务清单和数量检查。

默认模型建议：

| 任务类型 | 默认模型 |
|---|---|
| 图片生成 | 4.7 模型 2k 版 |
| 图片改图 / 参考图变体 | 4.7 模型 2k 版 |
| 视频生成 / 视频分段 | `seedance2.0fast_vip` |

用户明确指定模型版本时，按用户指定写入 `model` 字段。

## 5. 分批执行

大批量任务应分批，而不是一次性把所有任务塞给工具。

建议：

- 先完成第 1 批，检查身份锁和风格是否正确。
- 如果第 1 批通过，再继续后续批次。
- 每批完成后更新 manifest 状态。
- 失败项进入重试队列，不挤占未开始任务的编号。

默认批次：

| 总量 | 建议批次 |
|---|---|
| 1-6 | 一批 |
| 7-20 | 每批 4-6 |
| 21-60 | 每批 6-10 |
| 视频分段 | 每批 2-4 |

## 6. 重试策略

失败项必须保留原 `job_id`。

重试时：

- 不改编号。
- 说明失败原因。
- 增强对应约束。
- 最多重试 2-3 次，按技能质量标准决定。
- 重试仍失败时标为 `failed`，计入 `missing_count`。

## 7. 最终汇总

交付时必须包含：

```Markdown
## 执行汇总
- requested_count:
- planned_count:
- done_count:
- failed_count:
- missing_count:
- concurrency:
- batch_size:

## 任务清单
| job_id | target | tool | model | depends_on | status | retry_count | quality_gate |
|---|---|---|---|---|---|---|---|
```

如果任务链有最终合成步骤，例如电商视频或脱口秀视频，最终合成任务必须单独列为一个 `job_id`，并依赖所有必要上游素材。
